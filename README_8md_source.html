1 # Infrared4Arduino
2 Infrared4Arduino is an object-oriented infrared library for the Arduino.
3 
4 The library was (with some exceptions detailed below) written from scratch by myself.
5 It uses Michael Dreher's
6 IrWidget [(article in
7 German)](http://www.mikrocontroller.net/articles/High-Speed_capture_mit_ATmega_Timer),
8 see also [this forum
9 contribution](http://www.hifi-remote.com/forums/viewtopic.php?p=111876#111876).
10 
11 The main inspiration for the present work was the well-known library called IRremote, published by
12 Ken Shirriff in [his blog](http://www.righto.com/2009/08/multi-protocol-infrared-remote-library.html),
13 now maintained [on GitHub](https://github.com/z3t0/Arduino-IRremote). The file `boarddefs.h` has
14 essentially been stolen from IRremote.
15 [Chris Young's IRLib](http://tech.cyborg5.com/irlib/), ([GitHub repo](https://github.com/cyborg5/IRLib)),
16 should also be mentioned as a source of inspiration, but not of code.
17 
18 The classes `IrWidget` and `IrWidgetAggregating` are based on Michael's code.
19 The classes `IrReceiverSampler` and `IrSenderPwm`,
20 are adapted from Kevin's work. The remaining files are almost completely written
21 from scratch.
22 
23 This work is a low-level library (like `IRremote` and `IRLib`) that can be used in other projects,
24 like [AGirs](https://github.com/bengtmartensson/AGirs), which is an high-level program taking commands interactively
25 from a user or a program through a bi-directional `Stream`.
26 The goal has been to provide a sound, object oriented basis for the fundamental basis, not
27 to provide maximal functionality, the maximal number of protocols supported, or the most complete support of
28 different hardware. A clean design and high readability, without being "too" inefficient,
29 has been the highest priority. Dynamic memory allocation with `new` and `delete` is used extensively.
30 The user who is afraid of this can create his required objects at the start of the run, and keep
31 them. Most classes are immutable. The classes are `const`-correct.

38 ## Examples
39 A fairly large numbers of examples are provided. Most of these are a straight-forward

42 The examples `oppo_cooked` and `oppo_raw` consist of [IrScrutinizer](https://github.com/bengtmartensson/harctoolboxbundle)

47 ### MicroGirs
48 This "example" implements a [Girs](http://www.harctoolbox.org/Girs.html) server for the Arduino.
49 It can be used with [IrScrutinizer](https://github.com/bengtmartensson/harctoolboxbundle),
50 and [Lirc](http://lirc.org) (using the [Girs driver](http://lirc.org/html/girs.html)).
51 It is an interactive server that can send and receive IR signals, with some bells and whistles.
52 The interactive server is mainly meant for interact with other programs. communicating over a serial line, likely in USB disguise.
53 
54 MicroGirs is essentially a simplified version of [AGirs](https://github.com/bengtmartensson/AGirs),
55 stripped of features seldomly used (LCD and (visible-light) LED support, Ethernet, etc). As opposed to
56 AGirs, it does not depend on other libraries. For more documentation, see the AGirs project.
57 
58 There are a number of configuration options. These are all contained in the
59 file `config.h` and consists of CPP `#defines`.
60 
61 MicroGirs is essentially functionally equivalent to "GirsLite".

67 ## Types
68 There are some project specific data typedefs in `InfraredTypes.h`.
69 For durations in microseconds, the data type `microseconds_t` is to be
70 used. If desired/necessary, this can be either `uint16_t` or
71 `uint32_t`. For durations in milliseconds, use the type
72 `millisecons_t`. Likewise, use `frequency_t` for modulation frequency in
73 Hz (_not_ kHz as in the IRremote/IRLib).
74 
75 For "sizes", `size_t`, the standard C type, is used.
76 
77 Implementation dependent types like `int` are used if and _only if_ it is OK for the
78 compiler to select any implementation allowed by the C++ language.
79 
80 `unsigned int` is used for quantities that can "impossibly" be larger than 65535.
81 
82 ## IrSequences and IrSignals
83 An `IrSequence` is a vector of durations, i.e. sequence of interleaving gaps and spaces. It does not
84 contain the modulation frequence. As opposed to IRremote and IRLib, our sequences always start with
85 a space and end with a gap. It is claimed to be a more relevant representation than the one of IRremote and IRLib.
86 
87 An `IrSignal` consists of a modulation frequency and three `IrSequence`s: intro-, repeat-, and ending sequence. All of these, but not all, can be
88 empty. If repeat is empty, intro has to be non-empty and ending empty. The intro sequence is always sent first,
89 then comes a zero or more repeat sequences, and finally the ending sequence. To send a signal _n_ > 0 times shall mean
90 the following: If the intro is non-empty, send intro, _n_ - 1 repeats, and then the ending. If the intro is empty,
91 send _n_ repeats, and then then ending.

93 ## Class construction
94 For some receiving and transmitting classes, multiple instantiations are not sensible,
95 for other it may be. In this library, the classes that should only be instantiated
96 once are implemented as singleton classes, i.e. with no public constructor, but instead
97 a static "factory" method (`newThing()`) that delivers a pointer to a newly constructed instance
98 of `Thing`,
99 provided that `Thing` has not been instantiated before. The classes, where multiple instances
100 is sensible, come with public constructors. (However, the user still has to take responsibility
101 for avoiding pin- and timer-conflicts.)
102 
103 ## Hardware configuration
104 For hardware support, the file `IRremoteInt.h` from the IRremote project is used. This means that
105 all hardware that project supports is also supported here (for `IrReceiverSampler` and `IrSenderPwm`).
106 (Actually, a small fix, borrowed from IRLib, was used
107 to support Arduinos with ATMega32U4 (Leonardo, Micro).)
108 However, IrWidgetAggregating is currently supported on the boards Uno/Nano (ATmega328P), Leonardo/Micro (ATmega32U4),
109 and Mega2560 (ATmega2560).
110 
111 Several of the sending and receiving classes take a GPIO pin as argument to the constructor.
112 However, the sending pin of `IrSenderPwm` and the capture pin of `IrWidgetAggregating`
113 are not configurable, but (due to hardware limitations) have to be taken from the following table:
114 
115  Sender Pin Capture Pin
116  Uno/Nano (ATmega328P) 3 8
117  Leonardo/Micro (ATmega32U4) 9 4
118  Mega2560 (ATmega2560) 9 49

120 ## Timeouts
121 All the receiving classes adhere to the following conventions: When initialized, it waits
122 up to the time `beginningTimeout` for the first on-period. If not received within that period,
123 it returns with a timeout. Otherwise, is starts collecting data. It will collect data until one of the
124 following occurs:
125 * A silence of length `endingTimeout` has been detected. This is the normal ending. The detected last gap is returned with the data.
126 * The buffer gets full. Reception stops.

138 ## Error handling
139 Simple answer: there is none. If a function is sent erroneous data,
140 it just silently ignores the request, or does something else instead.
141 This (unfortunately) seems to be the standard procedure in Arduino programming.
142 
143 I am used to exception based error handling, for some reason this is not used by
144 the Arduino community.
145 
146 Constructive suggestions are welcome.

148 ## Protocols
149 Comparing with the predecessor works, this project may look meager, currently supporting only
150 two protocols (NEC1 and RC5). It is [planned](https://github.com/bengtmartensson/IrpTransmogrifier)
151 to generate the corresponding C++ code automatically from the IRP notation. (For this reason,
152 contributed implementations of more protocols are not solicited.)

154 ## Sending non-modulated signals.
155 RF signals (433 MHz and other carrier frequencies) do not use the IR
156 typical modulation. Also there are a few IR protocols (like [Revox, Barco,
157 Archer](http://www.hifi-remote.com/forums/viewtopic.php?t=14186&start=40))
158 not using modulation. These signals can be sent by the class `IrSenderNonMod`,
159 after connecting suitable hardware capable of sending non-modulated (IR- _or_ RF-) signals
160 to the GPIO pin given as argument to the constructor.

166 ## Questions and answers
167 
168 * What is the difference between the `IrReceiver*` and the `IrWidget*` classes?
169 
170 They are intended for two different use cases,
171 [receiving](http://www.harctoolbox.org/Glossary.html#ReceivingIrSignals)
172 and [capturing](http://www.harctoolbox.org/Glossary.html#Capturing).
173 Differently put, "receive" uses a
174 demodulating receiver (TSOPxxx, etc.),
175 "capture" a non-demodulating decoder (TSMPxxx, OPLxxx QSExxx, etc.). Note that
176 this terminology is not universally accepted (yet!).

178 # Coding style
179 
180 My goal is to write excellent code, even though I do not always succeed :-).
181 "Everything as simple as possible, but not simpler."
182 Cleanliness, logical structure, readability and maintainability are the most important
183 requirements. Efficiency (runtime and/or space) is also important, although it normally
184 comes on second place. [The Arduino Style Guide](https://www.arduino.cc/en/Reference/APIStyleGuide)
185 has different goals (essentially optimizing for novice programmers,
186 "Some of these run counter to professional programming practice"). It is therefore
187 not given priority in this project.

199 # Multi platform coding
200 For someone used to, e.g., Netbeans or Eclipse, the Arduino IDE
201 feels "somewhat" primitive and limited. In particular, it does not support debugging. Mainly
202 for this reason, the code in the present library is designed to compile, and at least
203 to some extent, run in a normal C++ environment on the host compiler. For this,
204 some code modifications, in particular, a customized `Arduino.h` was needed.
205 If the preprocessor symbol `ARDUINO` is defined,
206 just includes the standard Arduino `Arduino.h` is included, otherwise (i.e. for compiling for the host),
207 some more-or-less dummy stuff are defined, allowing compiling for, and execution/debugging
208 on the host.
209 
210 This way, certain types of problems can be solved much faster. The drawback is that the code
211 is "polluted" with ugly `#ifdef ARDUINO` statements, which decreases readability and
212 makes maintenance harder.
213 
214 The subdirectory `tests` contains test(s) that run on the host. The supplied `Makefile`
215 is intended for compiling for the host as target. It creates a library in the
216 standard sense (`*.a`), and can be used to build and run tests in subdirectory `tests`.
217 
218 With the provided `Doxyfile`, Doxygen will document only the (strict) Arduino parts,
219 not the "portable C++".
